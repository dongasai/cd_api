# 渠道亲和性设计

## 一、功能概述

### 1.1 核心概念

渠道亲和性（Channel Affinity）是一个**智能路由**功能，其核心目的是**让相同的请求（基于特定 Key）优先路由到之前成功使用过的渠道**，实现"粘滞"效果。

### 1.2 应用场景

- **CLI 工具连续对话**：如 Codex CLI、Claude CLI 等需要保持上下文一致性的场景
- **会话保持**：同一用户的连续请求路由到同一渠道，避免上下文丢失
- **Header 透传**：支持特定客户端的自定义 Header 需求
- **故障隔离**：可配置失败后不重试，快速返回错误

### 1.3 功能特性

1. **粘滞选路**：基于从请求上下文或 JSON Body 提取的 Key，优先复用上一次成功的渠道
2. **规则匹配**：支持按模型正则、路径正则、User-Agent 等条件匹配规则
3. **Key 来源**：支持从请求头、请求体、JSON Path 等多种来源提取亲和 Key
4. **参数覆盖**：命中规则后可自动合并参数覆盖模板
5. **缓存统计**：支持查看缓存命中率、使用统计等
6. **失败后不重试**：可配置某些规则命中后失败不重试

```sql
CREATE TABLE `channel_affinity_rules` (
    `id` bigint unsigned NOT NULL AUTO_INCREMENT,

    -- 高级配置

    `include_group_in_key` tinyint(1) DEFAULT 0 COMMENT '是否包含分组在 cache key 中',
    
    -- 状态
    `is_enabled` tinyint(1) DEFAULT 1 COMMENT '是否启用',
    `priority` int unsigned DEFAULT 0 COMMENT '优先级（数值越大越优先）',
    
    -- 元数据
    `hit_count` bigint unsigned DEFAULT 0 COMMENT '命中次数统计',
    `last_hit_at` timestamp NULL COMMENT '最后命中时间',
    
    `created_at` timestamp NULL DEFAULT NULL,
    `updated_at` timestamp NULL DEFAULT NULL,
    `deleted_at` timestamp NULL DEFAULT NULL,
    
    PRIMARY KEY (`id`),
    INDEX `idx_enabled_priority` (`is_enabled`, `priority` DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='渠道亲和性规则表';
```

### 2.3 缓存结构设计

使用 Laravel Cache 存储，Key 格式：

```
channel_affinity:{rule_id}:{key_hash}
```

#### key_hash 生成规则

`key_hash` 是根据规则配置的 `key_sources` 提取的值生成的唯一标识：

1. **提取阶段**：按 `key_sources` 配置顺序，从请求中提取对应的值
2. **组合阶段**：根据 `key_combine_strategy` 策略组合多个值
3. **哈希阶段**：对组合后的字符串进行 MD5 哈希，取前 16 位

**生成示例**：

| key_sources | combine_strategy | 提取值 | 组合结果 | key_hash |
|-------------|-----------------|--------|----------|----------|
| `[{"type": "api_key"}]` | `first` | `sk-abc123...` | `sk-abc123...` | `a1b2c3d4e5f6g7h8` |
| `[{"type": "api_key"}, {"type": "header", "key": "user-agent"}]` | `concat` | `sk-abc123...`, `RooCode/3.51.1` | `sk-abc123...\|RooCode/3.51.1` | `b2c3d4e5f6g7h8i9` |
| `[{"type": "api_key"}, {"type": "header", "key": "user-agent"}]` | `hash` | `sk-abc123...`, `RooCode/3.51.1` | `sk-abc123...\|RooCode/3.51.1` | `c3d4e5f6g7h8i9j0` |
| `[{"type": "client_ip"}]` | `first` | `192.168.1.100` | `192.168.1.100` | `d4e5f6g7h8i9j0k1` |

**代码实现**：

```php
public function generateKeyHash(array $values, string $strategy): string
{
    $combined = match ($strategy) {
        'first' => $values[0] ?? '',
        'concat' => implode('|', $values),
        'hash' => implode('|', $values),
        default => $values[0] ?? '',
    };
    
    return substr(md5($combined), 0, 16);
}
```

#### KeyExtractor

```php
<?php

namespace App\Services\ChannelAffinity;

use Illuminate\Http\Request;

class KeyExtractor
{
    /**
     * 从请求中提取亲和 Key
     * 
     * @param Request $request 请求对象
     * @param array $keySources Key 来源配置
     * @param string $combineStrategy 组合策略: first, concat, hash
     * @return string|null
     */
    public function extract(Request $request, array $keySources, string $combineStrategy = 'first'): ?string;
    
    /**
     * 从请求头提取
     */
    protected function extractFromHeader(Request $request, string $key): ?string;
    
    /**
     * 从 JSON Body 提取（支持 JSON Path）
     */
    protected function extractFromJsonPath(Request $request, string $path): ?string;
    
    /**
     * 从查询参数提取
     */
    protected function extractFromQuery(Request $request, string $key): ?string;
    
    /**
     * 获取 API Key（从认证信息）
     */
    protected function extractApiKey(Request $request): ?string;
    
    /**
     * 获取客户端 IP
     */
    protected function extractClientIp(Request $request): ?string;
    
    /**
     * 获取 User-Agent
     */
    protected function extractUserAgent(Request $request): ?string;
    
    /**
     * 组合多个 Key 值
     */
    protected function combineKeys(array $values, string $strategy): string;
    
    /**
     * 生成 Key 指纹（用于日志脱敏）
     */
    public function fingerprint(?string $key): string;
}
```

### 4.4 RooCode 亲和性配置示例

RooCode 是一个 VS Code 扩展，其请求特点如下：

**请求特征分析**：

| 字段 | 来源 | 示例值 | 说明 |
|------|------|--------|------|
| `user-agent` | Header | `RooCode/3.51.1` | 可识别客户端类型和版本 |
| `x-title` | Header | `Roo Code` | 客户端标识 |
| `http-referer` | Header | `https://github.com/RooVetGit/Roo-Cline` | 来源页面 |
| `model` | Body | `deepseek-chat` | 请求模型 |
| API Key | 认证 | `sk-xxx...` | 用户身份标识 |

**注意**：RooCode 请求中**没有**明确的会话 ID（如 `session_id`、`conversation_id`），因此需要使用组合策略。

#### 方案一：基于 API Key + User-Agent 组合亲和（推荐）

更精细的亲和策略，区分同一 API Key 下的不同客户端：

```json
{
    "name": "roocode-combined-affinity",
    "description": "RooCode 基于 API Key + User-Agent 组合的渠道亲和性",
    "model_patterns": [".*"],
    "user_agent_patterns": ["RooCode"],
    "key_sources": [
        {"type": "api_key"},
        {"type": "header", "key": "user-agent"}
    ],
    "key_combine_strategy": "hash",
    "ttl_seconds": 120,
    "skip_retry_on_failure": false,
    "include_group_in_key": true,
    "is_enabled": true,
    "priority": 100
}
```

**缓存 Key 格式**：`channel_affinity:{rule_id}:{key_hash}`

#### 方案二：基于客户端 IP 亲和

同一 IP 的 RooCode 请求路由到同一渠道（适用于固定 IP 用户）：

```json
{
    "name": "roocode-ip-affinity",
    "description": "RooCode 基于客户端 IP 的渠道亲和性",
    "model_patterns": [".*"],
    "user_agent_patterns": ["RooCode"],
    "key_sources": [
        {"type": "client_ip"}
    ],
    "ttl_seconds": 120,
    "skip_retry_on_failure": false,
    "include_group_in_key": true,
    "is_enabled": true,
    "priority": 90
}
```

#### Key 来源类型扩展

为支持 RooCode 等客户端，需要扩展 `key_sources` 支持的类型：

| type | 说明 | 示例 |
|------|------|------|
| `header` | 从请求头提取 | `{"type": "header", "key": "x-session-id"}` |
| `json_path` | 从 JSON Body 提取 | `{"type": "json_path", "path": "metadata.user_id"}` |
| `query` | 从查询参数提取 | `{"type": "query", "key": "session_id"}` |
| `api_key` | 使用认证的 API Key | `{"type": "api_key"}` |
| `client_ip` | 使用客户端 IP | `{"type": "client_ip"}` |
| `user_agent` | 使用 User-Agent | `{"type": "user_agent"}` |

#### 组合策略配置

当 `key_sources` 包含多个来源时，通过 `key_combine_strategy` 指定组合方式：

| 策略 | 说明 | 结果示例 |
|------|------|----------|
| `first` | 使用第一个非空值（默认） | `sk-xxx...` |
| `concat` | 拼接所有值 | `sk-xxx...|RooCode/3.51.1` |
| `hash` | 对组合值进行 MD5 哈希 | `a1b2c3d4e5f6...` |

## 七、注意事项

### 7.1 性能考虑

- 规则匹配使用缓存，避免每次请求都查询数据库
- Key 提取操作应尽量简单高效
- 缓存使用 Redis，设置合理的过期时间

### 7.2 安全考虑

- Key 指纹化存储，避免敏感信息泄露
- 参数覆盖模板需要验证，防止注入攻击
- 管理接口需要权限控制

### 7.3 兼容性

- 功能默认关闭，不影响现有行为
- 规则匹配失败时回退到原有选路逻辑
- 支持渐进式启用
