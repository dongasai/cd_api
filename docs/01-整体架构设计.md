# 整体架构设计

## 一、系统概述

AI 代理系统是一个统一的 AI 服务网关，提供以下核心能力：

- **统一接口**：一个 API 端点接入所有 AI 服务
- **协议转换**：OpenAI 与 Anthropic 格式互转
- **智能路由**：多渠道负载均衡、故障自动切换
- **安全管控**：API Key 权限管理、模型访问控制
- **数据洞察**：实时数据看板、用量统计、成本分析
- **Coding 管理**：支持 Coding 账户配额管理、自动禁用/启用渠道
- **渠道管理**：渠道分组、标签、模型映射等灵活配置

## 四、请求处理流程

```
1. 请求到达 Nginx
   │
   ▼
2. Laravel 接收请求
   │
   ▼
3. 中间件管道处理
   │
   ├─── CORS 中间件: 处理跨域请求
   │
   ├─── Authenticate 中间件:
   │    ├── 提取 Authorization Header
   │    ├── 解析 ApiKey
   │    ├── 查询 ApiKey 记录
   │    ├── 验证 ApiKey 状态和过期时间
   │    └── 设置当前用户上下文
   │
   ├─── RateLimiter 中间件:
   │    ├── 构建限流 Key
   │    ├── 检查 Redis 计数器
   │    └── 设置 RateLimit Headers
   │
   ├─── CheckCodingQuota 中间件:
   │    ├── 检查渠道绑定的 Coding 账户
   │    ├── 验证配额是否充足
   │    └── 自动禁用/启用渠道
   │
   └─── AuditLogger 中间件:
        ├── 请求阶段:
        │    ├── 生成 request_id (UUID)
        │    ├── 记录请求开始时间
        │    └── 提取请求元数据 (IP, User-Agent)
        └── 响应阶段:
             └── 注入 request_id 到响应头 (X-Request-Id)
        注: 所有日志记录由 ProxyServer 统一处理，详见"内部分发流程"
   │
   ▼
4. 路由匹配 → 控制器处理(组装数据)
   │
   ▼
5. 内部分发流程 (详见下方"内部分发流程"章节)
   │
   ├─────────────────────────────────────────┐
   │                                         │
   ▼                                         ▼
【非流式请求】                           【流式请求】
   │                                         │
   ▼                                         ▼
6. 返回给控制器                          6. 直接流式输出
   │                                    (SSE Stream)
   ▼
7. 控制器输出响应
   │
   ▼
8. 返回响应给客户端
```

### 5.1 核心服务说明

| 服务 | 职责 |
|------|------|
| **ProxyServer** | 代理服务入口，协调整个请求处理流程，统一管理所有日志记录 |
| **ChannelRouterService** | 渠道路由服务，根据模型选择最优渠道，支持负载均衡和故障转移 |
| **LoadBalancer** | 负载均衡算法实现，支持加权轮询、最少连接、响应时间优先 |
| **FailoverStrategy** | 故障转移策略，渠道不可用时自动切换到备用渠道 |
| **RetryStrategy** | 重试策略，支持指数退避重试，区分可重试/不可重试错误 |
| **CircuitBreaker** | 熔断器，防止故障扩散，支持熔断/半开/关闭三种状态 |
| **ProtocolConverter** | 协议转换入口，提供请求/响应/流式事件的转换方法 |
| **DriverManager** | 管理协议驱动，支持 OpenAI 和 Anthropic 协议 |
| **ProviderManager** | 管理 Provider 实例，根据渠道类型获取对应 Provider |
| **ChannelCodingStatusService** | 渠道 Coding 状态管理，支持自动禁用/启用 |

### 5.2 数据传输对象 (DTO)

| DTO | 用途 |
|-----|------|
| **StandardRequest** | 标准化请求格式，包含消息、模型、参数等 |
| **StandardResponse** | 标准化响应格式，包含内容、用量等 |
| **StandardStreamEvent** | 标准化流式事件 |
| **ProviderRequest** | Provider 层请求对象 |
| **ProviderResponse** | Provider 层响应对象 |
| **TokenUsage** | Token 使用量统计 |

```
│   (请求日志)         │          │     (响应日志)               │

│ audit_log_id        │          │ audit_log_id                │
│ request_id          │          │ request_id                  │
│ method, path        │          │ request_log_id              │
│ headers             │          │ status_code, status_message │
│ body_text           │          │ headers                     │
│ model, model_params │          │ body_text, body_binary      │
│ messages, prompt    │          │ response_type, finish_reason│
│ sensitive_fields    │          │ generated_text, usage       │
└─────────────────────┘          │ error_type, error_message   │
                                 │ upstream_provider, model     │
                                 └─────────────────────────────┘
```

### 5.5 日志创建时机

| 日志 | 创建时机 | 内容 | 存储方式 | 说明 |
|------|----------|------|----------|------|
| **RequestLog** | 请求到达 ProxyServer 时 | HTTP方法、路径、请求头、请求体、模型参数、客户端信息 | 同步写入 | 需立即获取 request_id 用于追踪 |
| **AuditLog** | 请求完成后 | 用户、API密钥、渠道、Token使用量、成本、延迟 | 同步写入 | 核心审计数据，需保证一致性 |
| **ResponseLog** | 请求完成后 | 状态码、响应体、生成内容、错误信息 | 同步写入 | 完整响应记录，便于问题排查 |

注: 所有日志均由 ProxyServer 统一同步写入，确保数据一致性和完整性。
