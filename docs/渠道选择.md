# 渠道选择机制

## 概述

渠道选择是 CdApi 代理系统的核心功能,负责根据请求参数从可用渠道池中选择最合适的渠道进行转发。选择过程遵循多阶段过滤和负载均衡策略,确保请求能够高效、稳定地路由到最优渠道。

## 核心流程

```
客户端请求
    ↓
ChannelSelector::select()
    ↓
渠道亲和性检查 (可选)
    ↓ (未命中)
ChannelRouterService::selectChannel()
    ↓
获取可用渠道列表
    ↓
API Key 渠道限制过滤
    ↓
透传协议匹配过滤
    ↓
User-Agent 限制过滤
    ↓
排除失败渠道
    ↓
负载均衡选择
    ↓
返回最终渠道
```

## 选择阶段详解

### 1. 渠道亲和性检查

**位置**: `ChannelSelector::select()` (第 36-75 行)

**触发条件**:
- 渠道亲和性功能已启用
- 不是故障转移重试请求 (failedChannels 为空)
- 请求匹配亲和性规则

**流程**:

```php
// 1. 查找匹配的亲和性规则
$affinityResult = $this->affinityService->getPreferredChannel(
    request(),
    $model,
    $group
);

// 2. 检查是否命中
if ($affinityResult->isHit && $affinityResult->channel) {
    // 3. 检查透传协议兼容性
    if ($affinityResult->channel->shouldPassthroughBody()) {
        $channelProtocol = $this->getChannelProtocol($affinityResult->channel);
        if ($channelProtocol !== $sourceProtocol) {
            // 协议不匹配,放弃亲和性渠道
            Log::warning('Affinity channel excluded due to passthrough protocol mismatch');
        } else {
            return $affinityResult->channel;
        }
    } else {
        return $affinityResult->channel;
    }
}
```

**亲和性匹配因素**:
- HTTP Header 值
- JSON 请求体路径值
- Query 参数
- API Key
- 客户端 IP
- User-Agent

**优势**:
- 相同来源的请求路由到同一渠道
- 避免协议转换开销
- 提升缓存命中率

**相关文件**:
- [ChannelSelector.php](../laravel/app/Services/Router/ChannelSelector.php)
- [ChannelAffinityService.php](../laravel/app/Services/ChannelAffinity/ChannelAffinityService.php)

### 2. 获取可用渠道列表

**位置**: `ChannelRouterService::getAvailableChannels()` (第 121-138 行)

**逻辑**:

```php
// 1. 从缓存获取(减少数据库查询)
$cacheKey = "channels:available:{$model}";
return Cache::remember($cacheKey, $this->config['cache_ttl'], function () use ($model) {
    // 2. 查询支持该模型的渠道
    $channelIds = $this->getChannelIdsForModel($model);

    // 3. 筛选活跃渠道
    return Channel::whereIn('id', $channelIds)
        ->where('status', 'active')
        ->orderBy('priority', 'desc')  // 优先级高的排前面
        ->orderBy('weight', 'desc')    // 权重大的排前面
        ->get();
});
```

**筛选条件**:
- 渠道状态为 `active`
- 渠道支持请求的模型 (通过 `channel_models` 表关联)
- 按 `priority` (优先级) 和 `weight` (权重) 排序

**缓存策略**:
- 缓存时间: 60 秒 (可配置)
- 缓存键: `channels:available:{model}`

**异常情况**:
- 如果没有可用渠道,抛出 `NotFoundHttpException` (模型不存在)

### 3. API Key 渠道限制过滤

**位置**: `ChannelRouterService::applyApiKeyChannelRestrictions()` (第 142-170 行)

**目的**: 根据 API Key 的黑名单/白名单配置过滤渠道

**过滤逻辑**:

```php
// 黑名单过滤
if ($apiKey->hasChannelBlacklist()) {
    $notAllowedChannels = $apiKey->getNotAllowedChannelIds();
    $channels = $channels->reject(function ($channel) use ($notAllowedChannels) {
        return in_array($channel->id, $notAllowedChannels, true);
    });
}

// 白名单过滤
if ($apiKey->hasChannelWhitelist()) {
    $allowedChannels = $apiKey->getAllowedChannelIds();
    $channels = $channels->filter(function ($channel) use ($allowedChannels) {
        return in_array($channel->id, $allowedChannels, true);
    });
}
```

**优先级**: 白名单 > 黑名单

**异常情况**:
- 如果过滤后无可用渠道,抛出 `BadRequestHttpException`

### 4. 透传协议匹配过滤

**位置**: `ChannelRouterService::applyPassthroughProtocolFilter()` (第 173-206 行)

**触发条件**: 渠道开启了 `body_passthrough` (请求体透传)

**逻辑**:

```php
return $channels->filter(function ($channel) use ($sourceProtocol) {
    // 未开启透传: 允许选择(会进行协议转换)
    if (!$channel->shouldPassthroughBody()) {
        return true;
    }

    // 开启透传: 检查协议是否匹配
    $channelProtocol = $this->getChannelProtocol($channel);
    return ($channelProtocol === $sourceProtocol);
});
```

**协议识别**:
- `anthropic` 协议: provider 为 `anthropic` 或 `claude`
- `openai` 协议: 其他 provider

**应用场景**:
- Anthropic 渠道开启透传后,只能处理 Anthropic 格式的请求
- OpenAI 渠道开启透传后,只能处理 OpenAI 格式的请求

**优势**:
- 减少协议转换开销
- 保持请求原始格式
- 提升性能

**异常情况**:
- 如果过滤后无可用渠道,抛出 `BadRequestHttpException`

### 5. User-Agent 限制过滤

**位置**: `ChannelRouterService::applyUserAgentFilter()` (第 208-227 行)

**触发条件**:
- 请求携带 User-Agent Header
- 渠道配置了 User-Agent 限制规则

**逻辑**:

```php
// User-Agent 为空时不过滤
if (empty($userAgent)) {
    return $channels;
}

// 调用 UserAgentFilterService 进行过滤
$userAgentFilter = app(UserAgentFilterService::class);
return $userAgentFilter->filterChannels($channels, $userAgent);
```

**过滤流程** (`UserAgentFilterService::filterChannels()`):

```php
return $channels->filter(function (Channel $channel) use ($userAgent) {
    $allowed = $channel->isUserAgentAllowed($userAgent);

    if (!$allowed) {
        Log::info('渠道User-Agent不匹配，已跳过', [
            'channel_id' => $channel->id,
            'channel_name' => $channel->name,
            'user_agent' => $userAgent,
        ]);
    }

    return $allowed;
});
```

**匹配规则**:
- 每个渠道可关联多个 User-Agent 规则
- 每个规则支持多个正则表达式 (JSON 数组存储)
- **任意一条正则匹配即允许访问**

**性能优化**:
- `has_user_agent_restriction` 字段快速筛选有限制的渠道
- 只对有限制的渠道执行正则匹配
- 无限制渠道直接通过

**安全考虑**:
- 正则表达式验证 (防止无效正则)
- ReDoS 防护 (检测危险模式)
- 错误处理 (无效正则跳过并记录日志)

**相关文件**:
- [UserAgentFilterService.php](../laravel/app/Services/Router/UserAgentFilterService.php)
- [UserAgent.php](../laravel/app/Models/UserAgent.php)

### 6. 排除失败渠道

**位置**: `ChannelRouterService::selectChannel()` (第 87-98 行)

**目的**: 在故障转移场景中,排除已经失败的渠道

**逻辑**:

```php
$excludeChannels = $context['exclude_channels'] ?? [];
if (!empty($excludeChannels)) {
    $channels = $channels->reject(function ($channel) use ($excludeChannels) {
        return in_array($channel->id, $excludeChannels, true);
    });
}
```

**应用场景**:
- 渠道请求失败后,系统会自动重试其他渠道
- 重试时排除已失败的渠道,避免重复失败

**异常情况**:
- 如果所有渠道都被排除,抛出 `ServiceUnavailableHttpException`

### 7. 负载均衡选择

**位置**: `ChannelRouterService::applyLoadBalancing()` (第 260-276 行)

**支持算法**:

#### 7.1 加权轮询 (Weighted Round Robin) - 默认

```php
protected function weightedRoundRobinBalance($channels): Channel
{
    $totalWeight = $channels->sum('weight');
    $random = mt_rand(1, $totalWeight);

    $currentWeight = 0;
    foreach ($channels as $channel) {
        $currentWeight += $channel->weight;
        if ($random <= $currentWeight) {
            return $channel;
        }
    }

    return $channels->first();
}
```

**特点**:
- 权重越高,被选中概率越大
- 基于随机数实现,避免热点
- 简单高效

**配置**: 在渠道的 `weight` 字段设置权重 (默认 1-100)

#### 7.2 最少连接数 (Least Connections)

```php
protected function leastConnectionsBalance($channels): Channel
{
    return $channels->sortBy('total_requests')->first();
}
```

**特点**:
- 选择当前请求数最少的渠道
- 适合长连接场景
- 需要实时统计请求数

**配置**: 通过 `total_requests` 字段统计

#### 7.3 响应时间优先 (Response Time)

```php
protected function responseTimeBalance($channels): Channel
{
    return $channels->sortBy('avg_latency_ms')->first();
}
```

**特点**:
- 选择平均响应时间最短的渠道
- 适合对延迟敏感的场景
- 需要实时统计延迟

**配置**: 通过 `avg_latency_ms` 字段统计

**算法选择**:

```php
$algorithm = $context['lb_algorithm'] ?? 'weighted_round_robin';
```

通过请求上下文的 `lb_algorithm` 参数指定算法。

## 故障转移机制

### 重试流程

**位置**: `ChannelRouterService::getFallbackChannelForRetry()` (第 344-367 行)

**逻辑**:

```php
public function getFallbackChannelForRetry(
    string $model,
    array $failedChannelIds,
    ?ApiKey $apiKey = null
): ?Channel {
    if (!$this->config['enable_failover']) {
        return null;
    }

    $channels = $this->getAvailableChannels($model);
    $channels = $this->applyApiKeyChannelRestrictions($channels, $apiKey);

    // 排除已失败的渠道
    return $channels->reject(function ($channel) use ($failedChannelIds) {
        return in_array($channel->id, $failedChannelIds, true);
    })->first();
}
```

**触发条件**:
- 配置启用故障转移 (`enable_failover = true`)
- 前一个渠道请求失败

**特点**:
- 自动排除失败渠道
- 应用相同的 API Key 限制
- 确保不会重复选择同一渠道

### 失败标记

**位置**: `ChannelRouterService::markChannelFailed()` (第 430-448 行)

```php
public function markChannelFailed(Channel $channel, string $reason = ''): void
{
    $channel->increment('failure_count');
    $channel->update([
        'last_failure_at' => now(),
    ]);

    Log::warning('Channel marked as failed', [
        'channel_id' => $channel->id,
        'channel_name' => $channel->name,
        'reason' => $reason,
        'failure_count' => $channel->failure_count,
    ]);
}
```

**记录内容**:
- 失败次数 (`failure_count`)
- 最后失败时间 (`last_failure_at`)

### 成功标记

**位置**: `ChannelRouterService::markChannelSuccess()` (第 450-461 行)

```php
public function markChannelSuccess(Channel $channel): void
{
    $channel->increment('success_count');
    $channel->update([
        'last_success_at' => now(),
    ]);
}
```

**记录内容**:
- 成功次数 (`success_count`)
- 最后成功时间 (`last_success_at`)

## 数据模型

### Channel 模型关键字段

| 字段 | 类型 | 说明 | 用于 |
|------|------|------|------|
| `status` | string | 渠道状态 (`active`/`inactive`) | 可用性筛选 |
| `priority` | int | 优先级 | 排序 |
| `weight` | int | 权重 (1-100) | 负载均衡 |
| `provider` | string | 供应商类型 | 协议识别 |
| `config` | array | 配置(JSON) | 透传设置等 |
| `has_user_agent_restriction` | boolean | 是否有UA限制 | 性能优化 |
| `failure_count` | int | 失败次数 | 统计 |
| `success_count` | int | 成功次数 | 统计 |
| `total_requests` | int | 总请求数 | 最少连接数算法 |
| `avg_latency_ms` | int | 平均延迟(ms) | 响应时间算法 |

### 关联关系

```
Channel
├── belongsToMany ChannelGroup (渠道分组)
├── belongsToMany UserAgent (UA规则,通过 channel_user_agent 表)
├── hasMany ChannelModel (支持的模型)

```

## 性能优化

### 1. 缓存策略

- **渠道列表缓存**: 60 秒 (减少数据库查询)
- **亲和性缓存**: 按 TTL 配置 (避免重复路由)

### 2. 快速筛选

- `has_user_agent_restriction` 标志快速筛选 UA 限制渠道
- `status` 索引快速筛选活跃渠道

### 3. 查询优化

- 使用 `whereIn` 批量查询
- 预加载关联关系 (`with`)
- 避免在循环中查询数据库

## 异常处理

| 异常类型 | 触发条件 | HTTP 状态码 |
|---------|---------|------------|
| `NotFoundHttpException` | 模型不存在 | 404 |
| `BadRequestHttpException` | API Key 限制/协议不匹配/UA不匹配 | 400 |
| `ServiceUnavailableHttpException` | 所有渠道失败 | 503 |

## 最佳实践

### 1. 权重配置

- 高性能渠道设置更高权重 (80-100)
- 普通渠道设置中等权重 (50-70)
- 备用渠道设置较低权重 (10-30)

### 2. 优先级使用

- 主要渠道设置高优先级 (100)
- 备用渠道设置低优先级 (50)
- 优先级 > 权重 (先按优先级筛选,再按权重负载均衡)

### 3. 透传协议

- 当上游供应商支持标准协议时,开启透传
- 避免不必要的协议转换
- 提升性能和兼容性

### 4. 渠道亲和性

- 适用于需要会话一致性的场景
- 避免因切换渠道导致的上下文丢失
- 提升缓存命中率

### 5. User-Agent 限制

- 用于限制特定客户端访问特定渠道
- 支持正则表达式灵活匹配
- 避免恶意客户端占用资源

## 监控指标

### 渠道健康度

- **成功率**: `success_count / total_requests`
- **平均延迟**: `avg_latency_ms`
- **失败次数**: `failure_count`
- **最后成功/失败时间**: `last_success_at` / `last_failure_at`

### 系统指标

- **缓存命中率**: 渠道列表缓存 / 亲和性缓存
- **选择耗时**: 路由选择总耗时
- **过滤比例**: 各阶段过滤掉的渠道数量
