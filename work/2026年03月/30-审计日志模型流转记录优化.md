# 审计日志模型流转记录优化

**日期**: 2026-03-30
**状态**: ✅ 已完成
**耗时**: 约2小时

## 背景

用户需求：完整记录模型名称在请求处理流程中的变化轨迹

**原有问题**：
- `actual_model` 字段语义不清晰（有时是渠道映射后的模型，有时是渠道响应的模型）
- 缺少中间过程数据（别名扩展、渠道匹配等）

## 需求

**字段规划**：
- `model` - 用户原始请求模型名（不变）
- `actual_model` - 渠道响应返回的模型名（语义变更）
- `apply_data` (新增) - 中间过程数据：
  - `matched_models`: 参与匹配的模型列表（别名扩展结果）
  - `channel_request_model`: 发送给渠道的模型名

**模型流转流程**：
```
用户请求模型 → Key映射 → 别名扩展(参与匹配模型) → 渠道映射(渠道请求模型) → 渠道响应模型
```

## 方案

采用**渐进式记录方案**：
- 选择渠道阶段记录 `apply_data`（ProxyServer）
- Handler 处理响应后更新 `actual_model`

**优点**：
- 职责清晰：数据在产生时立即记录
- 流程可追溯：完整记录每个关键环节
- 易于调试：出错时可快速定位

## 代码优化

**发现问题**：StreamHandler 中 `streamChunks` 重复遍历

**优化方案**：buildCompleteResponse 接收已提取的 `actualModel` 参数

**收益**：
- 减少 1 次数组遍历（流式响应可能包含数十个 chunks）
- 数据在源头提取，避免重复逻辑
- 向后兼容：可选参数

## 技术要点

1. **apply_data 数据来源**：
   - `matched_models`：调用 `ChannelRouterService::getModelNamesWithAliases()`
   - `channel_request_model`：调用 `ChannelRouterService::resolveModel()`

2. **actual_model 提取时机**：
   - 流式：从第一个 StreamChunk 的 model 字段提取
   - 非流式：从 ProtocolResponse 的 getModel() 方法获取

3. **向后兼容**：
   - apply_data 字段设置为 nullable
   - buildCompleteResponse 参数可选
   - 旧记录查询时需处理 null 值
