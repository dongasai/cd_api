# 请求错误分析

## 问题描述

从 2026-03-21 开始，系统出现大量流式请求失败，错误信息为：
```
Request body format invalid
```

错误代码：`InvalidParameter`

## 错误分析

### 1. 错误日志

```
[2026-03-22 17:54:02] local.ERROR: Provider error response
{"status":400,"body":{"error":{"code":"InvalidParameter","message":"Request body format invalid","param":null,"type":null}}}
```

### 2. 成功请求 vs 失败请求对比

| 项目 | 成功请求 (ID 1955) | 失败请求 (ID 2339) |
|------|-------------------|-------------------|
| 流式 (stream) | true | true |
| 工具数量 | 43 | 62 |
| max_tokens | 32000 | 32000 |
| thinking | {"budget_tokens":31999,"type":"enabled"} | {"budget_tokens":31999,"type":"enabled"} |
| output_config | {"effort":"medium"} | 不存在 |
| 请求体大小 | 144 KB | 120 KB |
| 时间 | 2026-03-15 20:23:57 | 2026-03-22 17:57:03 |

### 3. 新增的工具（34个）

失败请求中新增了以下工具：
- LSP
- mcp__plugin_claude-mem_mcp-search__* (8个)
- mcp__playwright__browser_* (21个)
- mcp__context7__* (2个)
- mcp__web-reader__webReader
- mcp__zhipu-search-prime__web_search_prime

### 4. URL 差异

**重要发现**：`full_url` 记录的是不带 queryString 的 URL，但 Provider 实际发送时会加上 queryString。

- 成功请求（ID 1956）记录的 full_url：`.../messages?beta=true`
- 成功请求（ID 1955）记录的 full_url：`.../messages`（无 beta，但原始请求有 beta=true）
- 失败请求（ID 2339）记录的 full_url：`.../messages`（原始请求有 beta=true）

这说明 `createChannelRequestLog` 方法记录 URL 时没有包含 queryString，但实际 Provider 发送时应该会加上。

### 5. ProxyServer.buildProviderRequest 分析

```php
// ProxyServer.php 第 352 和 368 行
$providerRequest->queryString = request()->getQueryString();
```

这段代码正确设置了 queryString，所以实际请求应该带有 `beta=true`。

## 可能原因

### 原因 1: 上游不支持某些工具

阿里云 Coding API 可能不支持某些新增的 MCP 工具定义，导致请求格式验证失败。

### 原因 2: 请求体过大

失败请求体约 120KB，虽然比成功请求小，但工具定义更复杂。

### 原因 3: thinking + tools 组合

当 `thinking` 参数和大量 `tools` 一起使用时，可能触发了上游的限制。

### 原因 4: output_config 参数

成功请求有 `output_config`，失败请求没有。这可能是关键差异。
